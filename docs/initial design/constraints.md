# 3. 제약 조건 (하지 말아야 할 것)

> 4단계 프레임 ③. 가이드라인이 되는 금지선. AI(구현자)와 사람 모두 이 선을 넘지 않는다.

---

## 3.1 핵심 제약 (절대선)

| # | 제약 | 이유 |
|---|---|---|
| C1 | **실제 LLM API를 호출하지 않는다.** 후보는 mock 데이터에서 생성. | 비용·키 회피. 데모 화면은 동일하게 보임 |
| C2 | **영속 DB(Postgres 등)를 쓰지 않는다.** 저장소는 인메모리(Upstash Redis / InMemory). | "메모리 DB" 제약. 데모용이라 영속 불필요 |
| C3 | **유료이거나 복잡한 인증이 필요한 외부 API를 붙이지 않는다.** 그런 게 필요하면 **무료·무인증 대체재를 찾거나, mock으로 대체**한다. | 형의 명시 제약. (예: 지도 API → 안 붙임, 후보는 텍스트만) |
| C4 | **가입·로그인 벽을 만들지 않는다.** 참여자는 닉네임만으로 입장. | 아하 모먼트(진입 마찰 0) 보호 |
| C5 | **카테고리를 음식점 1개로 고정.** 영화·여행·카페 등으로 넓히지 않는다. | 데모 한 동선에 집중 |

---

## 3.2 범위 제약 (이번 데모에서 안 만드는 기능)

- 순위 투표, 가중 집계
- AI 타이브레이크 / 후보 좁히기
- 개인화 추천, 과거 결정 이력 기반 추천
- 실제 지도·거리·영업시간 연동
- 소셜 로그인, 영구 그룹/친구/팔로우, 푸시 알림
- 예약·결제·주문·배달 연동
- 정밀 분석·대시보드 (재사용률 등 후행 지표)

> 위 항목은 PRD의 P1/P2다. **설계상 자리는 열어두되(포트·도메인 추상화), 이번엔 구현하지 않는다.**

---

## 3.3 기술 제약 / 가드레일

- **서버리스 전제를 위반하는 구현 금지** — 상시 WebSocket 서버, 인스턴스 로컬 메모리에 상태 저장(전역 변수에 방 보관) 등. 공유 상태는 반드시 Redis 어댑터를 통한다.
- **core는 인프라를 import 하지 않는다** — `core/` 안에서 `next/*`, `@upstash/*`, fetch 등 금지. 포트 인터페이스에만 의존.
- **비밀키는 코드에 하드코딩 금지** — Upstash 토큰 등은 env로만.
- **과도한 추상화 금지** — 데모다. 포트는 실제로 갈아끼울 3개(저장소·후보생성·시계) 정도에만 두고, 나머지는 단순하게.

---

## 3.4 "대체재를 찾아라"가 적용되는 지점

| 원래 PRD가 기대한 것 | 데모에서의 대체재 |
|---|---|
| LLM 후보 생성 (유료) | mock 후보 풀에서 조건 매칭해 5개 선택 + 인위적 지연 연출 |
| 카카오/네이버 지도 API (키·쿼터) | 후보를 텍스트 카드로만 표시 (좌표·거리 없음) |
| 영속 DB | Upstash Redis 무료 티어 (카드 X) / 로컬은 InMemory |
| 실시간 서비스(Pusher 등, 키 발급) | 1초 폴링 (외부 서비스 0) |
| QR 코드 스캐너/생성 인프라 | 그냥 공유 **링크** (원하면 클라이언트에서 무료 QR 라이브러리로 표시 가능, 선택) |
