# 1. 기능 요구사항 (무엇을 만들 것인가)

> 4단계 프레임 ①. "무엇을 만들고, 내가 무엇을 원하는지" 명확히 하는 단계.
>
> ⚠️ **이 문서는 초기 설계 스냅샷이다.** 세부 설계에서 **타이머/카운트다운**, **방장도 투표자**, **대기 라운지**, **URL 복사 공유**가 흐름에 추가됐다. 최신 흐름·화면은 `../detailed design/detailed-design.md` (③④) 참조.

---

## 1.1 한 줄 정의

방을 만들면 후보가 자동으로 뜨고, 사람들이 **링크로 가입 없이 입장**해 **실시간으로 같이 투표**해서 **3분 안에** 점심 식당을 정하는 데모.

**이 빌드의 목적은 실서비스가 아니라 회의에서 컨셉을 직접 눌러보게 하는 것이다.** 따라서 "진짜로 동작하는 핵심 동선"을 우선하고, 깊이(개인화·지도·예약 등)는 전부 뒤로 미룬다.

---

## 1.2 풀려는 문제

2~8인 그룹이 무언가 정할 때, 결정 자체보다 **후보를 모으고 의견을 합치는 과정**에서 시간이 샌다. "아무거나 / 다 좋아"가 반복되고, 늘 한 사람이 총대를 메 후보를 검색해야 투표가 시작된다.

모디가 없애는 두 가지 마찰:
1. **콜드 스타트** — 빈 화면에서 후보를 사람이 채우는 일 → 자동 후보 생성으로 해소
2. **진입 마찰** — 가입·로그인 벽 → 링크 + 닉네임만으로 즉시 입장

---

## 1.3 페르소나 (데모 시연 기준)

| 페르소나 | 역할 | 성공의 정의 |
|---|---|---|
| 🅰 **김총대** (방장) | 방을 만들고 조건을 넣는 사람 | 후보를 직접 안 찾아도 5개가 즉시 뜬다 |
| 🅱 **이아무** (참여자) | 링크로 들어와 투표하는 사람 | 가입 없이 10초 만에 입장해 "이건 빼줘"를 표시 |

> 데모 시연 시나리오 = 🅰가 방을 만들고 🅱들이 링크로 우르르 입장해 같이 투표하는 그림.
> 🅲(반복 모임 운영자·개인화)는 이 데모 범위 밖.

---

## 1.4 아하 모먼트 (데모에서 반드시 보여줄 두 순간)

- **아하 ① (방장):** 조건만 넣었는데 **후보 5개가 카드로 즉시** 뜬다 → "내가 검색 안 해도 되네."
- **아하 ② (참여자):** 링크를 눌렀더니 **가입창 없이 바로 투표 화면**, 표가 **실시간으로 차오른다** → "가입도 안 했는데 벌써 같이 정하고 있네."

설계 원칙: **아하 전에 가입·로그인 벽을 절대 두지 않는다.** (게스트 입장이 P0인 이유)

---

## 1.5 사용자 스토리 (P0, 우선순위 순)

1. **방장**으로서, 주제(점심)와 조건(위치·인원·예산)을 넣으면 **후보 식당 5곳이 자동으로 뜨길** 원한다. — 직접 검색·정리 노동 제거.
2. **참여자**로서, 초대 링크를 누르면 **로그인 없이 닉네임만으로 바로 입장해 투표**하고 싶다. — 진입 마찰 제거.
3. **참여자**로서, 다른 사람들 투표가 **실시간으로 차오르는 걸** 보고 싶다. — 분위기 읽고 빨리 합의.
4. **방장**으로서, 투표가 모이면 **결과를 확정**하고 싶다. — 깔끔한 마무리.
5. **(엣지) 참여자**로서, 후보가 다 별로면 **직접 후보를 추가**하고 싶다. — 빈틈 메우기.
6. **(엣지) 누구나**, **아무도 투표 안 했거나 1명만 들어온 빈 방**에서도 깨지지 않고 안내받고 싶다.

---

## 1.6 범위 (P0 — 데모에 반드시 들어가는 것)

| 기능 | 동작 |
|---|---|
| 방 생성·초대 | 주제·조건 입력 → 방 생성 → 공유 **링크** 발급. 게스트(닉네임만) 입장 허용 |
| 후보 생성 | 조건 기반 후보 5개 자동 표시 — **mock 데이터** (LLM 호출 없음, 약간의 지연으로 "생성 중" 연출) |
| 실시간 참여·투표 | 링크로 입장, 후보에 **좋아요 / 싫어요** |
| 실시간 집계·확정 | 표가 **1초 폴링**으로 갱신, 방장이 결과 확정 |
| 결정 기록 | 확정 결과를 (메모리) 저장 — 데모용 |

### 비범위 (이 데모에서 안 함)

- ❌ 실제 LLM 호출 (mock으로 대체)
- ❌ 실제 지도/식당 검증, 영업시간 (카카오·네이버 API) — 후보는 글 텍스트만
- ❌ 순위 투표, AI 타이브레이크, 개인화 추천
- ❌ 소셜 로그인, 영구 그룹/친구
- ❌ 예약·결제·배달 연동
- ❌ 다른 카테고리(영화·여행 등) — **음식점 1개 카테고리만**

> 자세한 금지선은 [constraints.md](constraints.md).

---

## 1.7 성공 기준 (데모 관점)

데모 워크스루에서 아래가 **실제로 동작하면 성공**이다.

- 방 생성 → 후보 5개 노출까지 **체감 3초 내**.
- 링크 클릭 → 닉네임 입력 → 투표까지 **가입 0회**.
- 한 기기에서 투표하면 **다른 기기 화면에 1~2초 내 반영**(폴링 주기).
- 방장이 결과를 확정하면 모든 참여자 화면에 결과가 표시된다.

> 정밀 분석 지표(재사용률 등)는 데모 범위 밖. "회의에서 눌러보면 된다"가 유일한 성공 척도.
