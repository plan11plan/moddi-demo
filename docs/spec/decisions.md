# 기술적 결정사항 — 모디

> 이 표가 우리의 개발 철학이다: **회의용 데모라는 목적에 맞춰, 속도와 단순함을 사고 안정성은 "라이브에서 깨질 곳"에만 쓴다. 외부 의존·비용·인증은 회피하되, 코어는 나중에 분리 가능하게 추상화해 둔다.**

| 영역 | 결정 | 왜 (의도/맥락) | 트레이드오프 → 재검토 트리거 |
|---|---|---|---|
| 형태 | Next.js 풀스택 + 헥사고날, Vercel | 데모를 가장 빠르게 배포. 단 core를 인프라서 분리해 **나중에 Spring 서버로 들어낼 여지** 보존(사용자=백엔드 트랙) | 하이브리드(별도 백엔드)는 지금 과함 |

| 신원 | localStorage(방ID별) | 무로그인이 제품 기둥(진입마찰0). 새로고침 글리치만 막으면 됨 | 기기 바뀌면 새 사람(데모 허용) |
| UI | Tailwind + shadcn, 웜 톤 | 카드 중심 모바일 데모를 빠르고 예쁘게. 레퍼런스 톤(에스프레소·크림·테라코타) | — |

## 제약 (절대선 — 넘으면 목적 위반)
1. 실제 LLM 호출 X(mock). 2. 영속 DB X(인메모리). 3. 유료·복잡인증 외부 API X(필요시 무료·무인증 대체 또는 mock). 4. 가입·로그인 벽 X(닉네임만). 5. 카테고리 음식점 1개 고정.

## 가드레일 (구현 중 지킬 것)
- core에서 `next/*`·`@upstash/*`·fetch 직접 호출 금지 — 포트에만 의존.
- 공유 상태는 반드시 Redis 어댑터 경유(인스턴스 로컬 전역변수에 방 저장 금지 — 서버리스서 안 공유됨).
- 비밀키는 env로만. 과도한 추상화 금지(포트는 저장소·후보·시계 3개로 충분).
