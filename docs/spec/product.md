# 제품 기능 정의서 — 모디(MODI)

**한 줄:** 방을 만들면 후보가 자동으로 뜨고, 사람들이 링크로 가입 없이 들어와 실시간으로 같이 투표해 3분 안에 점심 식당을 정한다.

**성격(가장 중요한 전제):** 이건 **스위프 4조 1주차 회의용 "보여주기식" 데모**다. 실서비스도, 9주 본프로젝트도 아니다. → 모든 결정은 "회의에서 실제로 눌러보게 한다"에 최적화하고, 깊이(개인화·지도·예약·확장성·영속성)는 의도적으로 버린다. **단순함 > 완성도.**

## 푸는 문제
2~8인이 점심을 정할 때 결정보다 **(1) 후보 모으기(콜드스타트)** 와 **(2) 의견 합치기**에서 시간이 샌다. 모디가 없애는 두 마찰:
- **콜드스타트** → 후보 자동 생성(데모는 mock).
- **진입 마찰** → 링크 + 닉네임만, 가입·로그인 없음.

## 핵심 가치 = 아하 모먼트 2개 (이걸 깨면 제품이 죽는다)
1. **방장:** 조건만 넣었는데 후보 5개가 즉시 뜬다 → "검색 안 해도 되네." (방 생성 후 ~3초)
2. **참여자:** 링크 눌렀더니 가입창 없이 바로 투표, 표가 실시간으로 차오른다 → "벌써 같이 정하네." (클릭 후 ~10초)
> 설계 철칙: **아하 전에는 어떤 벽(가입·로그인)도 두지 않는다.**

## 페르소나 (데모 시연 기준)
- **방장(김총대):** 방을 만들고 조건을 넣는 사람. 성공 = 직접 안 찾아도 후보가 뜬다. **방장도 투표자다.**
- **참여자(이아무):** 링크로 들어와 투표. 성공 = 가입 없이 입장해 "이건 빼줘(👎)"를 표시.

## 기능 (P0 — 데모에 반드시)
| 기능 | 동작 |
|---|---|
| 방 생성·초대 | 조건(모임명·위치·인원·예산·타이머) 입력 → 방 생성 → 공유 **링크** 발급 |
| 후보 생성 | 조건 기반 후보 5개 자동 — **mock**(LLM 호출 0), 1.5s "생성 중" 연출 |
| 게스트 입장 | 링크 + 닉네임만(가입 X) |
| 실시간 투표 | 후보마다 👍/👎(독립·토글), 표는 1초 폴링으로 차오름 |
| 타이머·확정 | 방장이 투표 시작 → 타이머 0 또는 "지금 마감" → 1등 자동 공개(축하 연출) |

세부 흐름·화면은 `user-flow.md`.

## 안 하는 것 (비목표 — 전부 "데모라 좁힘"이 이유)
실제 LLM 호출 / 실제 지도·식당 검증·영업시간 / 순위투표·AI 타이브레이크·개인화 / 소셜 로그인·영구 그룹 / 예약·결제·배달 / **음식점 외 카테고리**. → 설계상 자리만 열어두고(포트 추상화) 구현 안 함. 제약 근거는 `decisions.md`.

## 성공 기준
데모로 **브라우저에서 직접** 아래가 되면 성공(자동 테스트는 보조):
- 방 생성 → 후보 5개 ≤3초 · 링크 클릭 → 투표까지 가입 0회 · 한 기기 투표 → 다른 기기 1~2초 내 반영 · 방장 확정 → 전원 결과 표시.
