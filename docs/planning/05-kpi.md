# ⑤ KPI — 모디(MODI)

> 측정 철학: **North Star 하나로 방향을 모으고, HEART로 사용자 경험을, AARRR로 성장 퍼널을 본다.** 데모 부트캠프라 정밀 분석보다 **"보여줄 수 있는 숫자 + 검증 가능한 가설"** 우선.

## 1. North Star Metric (NSM)

> **NSM = 주간 "완료된 그룹 결정" 수 (Weekly Completed Decisions, WCD)**
> 정의: 한 주 동안 `status=decided`에 도달하고 **실투표 참여자 ≥ 2명**인 방의 수.

- **왜 이게 북극성인가:** 모디가 세상에 더하는 가치 = "그룹이 빠르게 한 곳에 합의함". 결정 1건 = 가치 1단위. 참여자 ≥2명 조건으로 "혼자 만든 빈 방"을 걸러 **진짜 그룹 합의**만 카운트.
- **NSM이 좋아지면 따라 좋아지는 것:** 활성화(후보·투표 잘 돌아야 결정됨), 참여(여러 명 투표), 리텐션(좋으면 또 결정), 추천(링크 공유로 인원 유입).
- **NSM을 헛되이 올리는 꼼수 차단:** 참여자 수·결정 소요시간 가드레일(아래)로 "질 낮은 결정 양산" 방지.

## 2. HEART 프레임워크 (Google) — 경험 품질

| 차원 | 우리 정의 | 핵심 지표(Signal) | 데모 목표 |
|---|---|---|---|
| **Happiness** | 결정 경험 만족 | 사후 설문 만족도(5점), "다시 쓸 의향" NPS-lite | 만족 ≥ 4.0 / 재사용의향 ≥ 60% |
| **Engagement** | 방 안 활발함 | 방당 실투표 참여자 수, 인당 투표 액션 수 | 방당 투표자 ≥ 3명 |
| **Adoption** | 핵심 기능 첫 사용 | 아하① 도달률, 아하② 도달률(②) | 98% / 60% |
| **Retention** | 다시 돌아옴 | 재사용률(같은 사용자/그룹 2회+), 7일 리텐션(P1) | 데모: 재사용 의향으로 대체 측정 |
| **Task success** | 결정까지 끝냄 | **결정 완료율**, **결정 소요시간** | 완료율 ≥ 80% / 소요 ≤ 3분 |

> Happiness·Retention은 데모 규모에선 행동 데이터가 적어 **설문(⑥)으로 보완**. 나머지는 이벤트 로깅으로 직접 계측.

## 3. AARRR 해적 지표 — 성장 퍼널

| 단계 | 지표 | 측정 | 데모 목표 |
|---|---|---|---|
| **Acquisition** | 방문/링크 오픈 | `link_opened`, 랜딩 방문 | — (데모 현장 유입) |
| **Activation** | 아하 도달 | 방생성→후보 98%, 링크→투표 60% | ② 참고 |
| **Retention** | 재사용 | 동일 사용자/그룹 2회+ | 후행, P1 본격화 |
| **Referral** | 링크 공유→입장 | `link_copied` → 신규 `joined` (방당 평균 입장 수) | 방당 입장 ≥ 3 |
| **Revenue** | (v1 비범위) | — | 데모 무료, 수익화는 로드맵 밖 |

## 4. 선행 vs 후행 지표

- **선행(데모에서 즉시 보임):** 결정 완료율 · 결정 소요시간 · 아하 도달률 · 방당 참여자 수 · 투표 참여율.
- **후행(있으면 강한 어필):** 재사용률 · 만족도/재사용 의향(설문).

## 5. 계측 설계 — 이벤트 스키마

> 백엔드 이벤트 로깅(타임스탬프) → 간단 집계. 아래 굵은 이벤트는 **현재 MVP 동작에 이미 존재**(방 상태·투표 시각). 일부는 데모용 로깅만 추가하면 됨.

| 이벤트 | 발생 시점 | 핵심 속성 | 쓰임(지표) |
|---|---|---|---|
| `room_created` | 방 생성 | roomId, location, headcount, budget, timerMinutes | 분모(방 수), 아하① 시작 |
| `candidates_shown` | 후보 노출 | roomId, count, latencyMs | 아하① 도달·TTV |
| `link_opened` | 초대 링크 진입 | roomId | 아하② 시작, Acquisition |
| `joined` | 닉네임 입장 | roomId, participantId | 방당 참여자, Referral |
| `vote_cast` | 투표(upsert) | roomId, candidateId, value, seq | Engagement, 아하② 도달(first) |
| `candidate_added` | 후보 직접 추가 | roomId, by | 참여 깊이 |
| `room_decided` | 확정(수동/타이머) | roomId, winnerId, voters, durationSec, byTimer | **NSM**, 완료율, 소요시간 |

**파생 지표 정의**
- 결정 완료율 = `room_decided` / `room_created`.
- 결정 소요시간 = `room_decided.ts − room_created.ts` (목표 중앙값 ≤ 3분).
- 아하① 도달률 = `candidates_shown` / `room_created`.
- 아하② 도달률 = first `vote_cast` 한 participant / `link_opened`.
- 방당 참여자 = `joined` distinct / 방.

## 6. 가드레일(역지표) — NSM을 망가뜨리지 않게

| 가드레일 | 임계 | 왜 |
|---|---|---|
| 결정 소요시간 | 중앙값 ≤ 3분 | 결정 수 늘리려다 느려지면 가치 훼손 |
| 표 유실/중복 | 0건 | 정합성 깨지면 "살아있는 결정" 신뢰 붕괴 (배포 시 폴링 write 버그 수정 완료) |
| 빈 방 비율 | 모니터링 | 참여자 1명 방 양산 방지(NSM 분자에서 제외) |
| 후보 노출 지연 | p90 ≤ 3초 | 아하① 사수 |

## 7. 데모데이 스코어보드 (발표 때 띄울 숫자)
현장 관객 참여형 데모라 **실시간으로 보여줄 수 있는 4개**:
1. **결정 완료율** (방 생성 → 확정 도달 %)
2. **결정 소요시간** (목표 3분 이내 — 라이브로 타이머)
3. **방당 참여자 수 / 투표 참여율**
4. **AI 후보 채택률** (최종 선택이 AI(mock) 후보였던 비율 — AI 가치 증명)

> 측정 방법: 데모 세션 동안 위 이벤트 집계 → 발표 슬라이드에 라이브/스냅샷 숫자. 정밀 대시보드보다 **"심사위원이 직접 만든 방의 숫자"** 가 설득력.

## 연결
- 활성화 목표(아하 도달률) = ②와 동일 정의.
- Happiness·Retention 목표치는 ⑥ 설문으로 측정/검증(H1·H2 가설).
- NSM·완료율은 ③ 사용 Flow의 `decided` 종착점과 ④ 저니맵의 "결정 확정" 단계에 대응.
