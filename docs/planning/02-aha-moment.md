# ② 아하 모먼트 — 모디(MODI)

> 아하 모먼트 = 사용자가 "어, 이거 되네"라고 **가치를 처음 체감하는 그 순간**. 활성화(Activation)의 정의이자 모든 온보딩·KPI의 기준점.
> 모디는 **역할이 둘**이라 아하도 **둘**이다. 둘 다 못 터지면 그룹 결정이 시작조차 안 된다.

## 1. 두 개의 아하

### 아하 ① — 방장: "내가 검색 안 해도 되네"
- **순간:** 위치·예산만 넣고 생성 버튼 → **3초 안에 '안 가본 집' 포함 후보 5개가 카드로** 뜬다.
- **체감 가치:** 콜드 스타트(빈 화면 후보 채우기) 노동이 사라짐. = 🅰 김총대의 JTBD 정통 타격.
- **TTV(Time-to-Value): ≤ 3초** (방 생성 → 첫 후보 노출).

### 아하 ② — 참여자: "가입도 안 했는데 벌써 같이 정하네"
- **순간:** 링크 클릭 → **가입창 없이 바로 투표 화면**, 내가 누른 표와 남의 표가 **실시간으로 차오름**.
- **체감 가치:** 진입 마찰 0 + "같이 정하는 중"이라는 실시간 소속감. = 🅱 이아무의 JTBD 정통 타격.
- **TTV: ≤ 10초** (링크 클릭 → 첫 투표).

| 페르소나 | 아하가 오는 지점 | 깨닫는 것 |
|---|---|---|
| 🅰 김총대 | 후보 5개 즉시 생성 | "후보 찾는 노동이 사라졌다" |
| 🅱 이아무 | 링크→가입 없이 투표 | "마찰 없이 내 의견(특히 싫은 것)을 냈다" |
| 🅲 박모임(P1) | 2회차에 "지난번과 다른" 추천 | "얘가 우리 모임을 기억한다" |

## 2. 아하를 지키는 설계 철칙 (Non-negotiables)
1. **아하 전에는 어떤 벽도 없다.** 가입·로그인·결제·튜토리얼 금지. (게스트 입장이 P0인 이유)
2. **아하①은 ≤3초, 아하②는 ≤10초.** 넘으면 이탈. 후보 생성 지연 연출도 1.5초 내.
3. **첫 화면이 곧 가치.** 방장은 후보 카드, 참여자는 투표 화면이 첫 화면.
4. **빈손 금지.** 후보 0개·표 유실은 아하를 즉사시킴 → mock 폴백 5개 보장, 투표 정합성 보장(배포 시 폴링 RMW 버그 수정 완료).

## 3. 측정 — "아하 도달률 = 활성화율"

핵심 퍼널 2개. 이 둘이 활성화의 전부.

| 아하 | 활성화 이벤트(시작→도달) | 목표 도달률 | TTV 목표 |
|---|---|---|---|
| ① 방장 | `room_created` → `candidates_shown` | **≥ 98%** (거의 항상 떠야) | p90 ≤ 3s |
| ② 참여자 | `link_opened` → `first_vote_cast` | **≥ 60%** (링크 연 사람 중 투표) | p90 ≤ 10s |

> 보조: `join`까지 갔는데 투표 안 한 비율(=관망), 아하②의 누수 지점.

## 4. 아하 → 습관 (Hook 모델로 본 재방문 설계)
- **Trigger:** "밥?" 단톡(외부 트리거) → (P1) 모디 알림(내부 트리거).
- **Action:** 방 생성 / 링크 클릭+투표 — 최소 행동.
- **Variable Reward:** 매번 다른 후보(셔플)·실시간으로 차오르는 표·결과 발표 연출 → 예측 불가의 재미.
- **Investment:** 결정 기록 저장(P0) → (P1) 개인화로 다음 결정이 더 좋아짐 → 🅲 리텐션 루프.

## 5. "매직 넘버" 가설 (검증 대상)
> Facebook "10일 내 친구 7명"처럼, 모디의 재방문을 예측하는 행동 임계치 가설:
- **가설 H1:** *첫 방에서 결정 완료(decided)까지 경험한 그룹은 재사용 확률이 유의하게 높다.*
- **가설 H2:** *방당 실투표 참여자 ≥3명이면 "살아있는 결정" 체감 → 만족·재사용↑.*
- → ⑤ KPI의 활성화/리텐션 목표와 ⑥ 설문 가설로 연결, 데모 후 데이터로 검증.

## 6. 안티-아하 (가치 체감을 죽이는 것 — 모니터링)
빈 후보 / 가입 벽 / 3초 초과 지연 / 표 유실·렉 / "누가 했는지 모름" / 동점인데 교착. → 각 항목을 ⑤의 가드레일 지표로 추적.
