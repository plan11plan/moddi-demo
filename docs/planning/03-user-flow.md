# ③ 기획서용 사용 Flow — 모디(MODI)

> 작성 관점: Product Manager + Product Designer.
> 이 문서는 스크린 스펙(→ `docs/spec/user-flow.md`)이 아닌 **의사결정 기반 Flow 기획서**다.
> "왜 이 순서인가, 이 분기에서 무슨 결정을 내렸는가"를 못 박아, ④ 저니맵·⑥ 설문·개발 백로그가 같은 근거 위에 서게 한다.

---

## 1. 핵심 사용 Flow (Happy Path)

두 페르소나(🅰 방장 / 🅱 참여자)의 레인을 병렬로 그린다.
각 단계 옆에 아하 발화 지점(②)과 KPI 이벤트(⑤)를 명시한다.

```
【 방장 🅰 레인 】                               【 참여자 🅱 레인 】
                                                (링크 수신 — 카톡/단톡)
랜딩 (/)                                                ↓
  ↓  [만들기 클릭]                              링크 클릭 → /room/[id]
생성 폼 (/create)                                        ↓
  모임명 · 위치 · 인원 · 예산 · 타이머                닉네임 입력 (가입 X)
  ↓  [방 생성]                                           ↓
1.5s "후보 뽑는 중" 연출                         ★ 아하② 시작 (link_opened)
  ↓                                              대기 라운지 진입 (joined)
★ 아하① 발화                                             ↓
후보 5개 카드 즉시 표시                          "방장이 곧 시작해요" 안내
(candidates_shown, TTV ≤3s)                      👥 N명 모이는 중 — 실시간↑
  ↓                                                       ↓
URL 복사 → 단톡 공유                        ┄┄┄┄┄┄ (방장 [투표 시작]) ┄┄┄┄┄
  ↓  [투표 시작] (room_voting_started)                     ↓
투표 화면 (voting)                              투표 화면 전환 (voting)
  타이머 가동 · 후보 👍/👎                      ★ 아하② 발화
  남의 표 1초 폴링으로 차오름                   첫 👍 또는 👎 클릭 (first_vote_cast)
  [지금 마감] 버튼 노출                          TTV ≤10s — (link_opened → first_vote_cast)
  ↓  (타이머 0 or [지금 마감])                   후보마다 독립 투표 (vote_cast × N)
결과 발표 (decided)                                        ↓
  🎉 컨페티 + 1등 카드 확정                      결과 발표 (decided)
  (room_decided)                                  🎉 공동 확인
  [처음으로] → 다음 결정                          [처음으로]
  ↓
NSM +1 카운트 (실투표자 ≥2명 조건 충족 시)
```

### 아하 발화 지점 요약 (②)

| 아하 | 발화 순간 | TTV 목표 | KPI 이벤트(⑤) |
|---|---|---|---|
| ① 방장 | 방 생성 → 후보 5개 카드 즉시 노출 | p90 ≤ 3s | `room_created` → `candidates_shown` |
| ② 참여자 | 링크 클릭 → 첫 투표 완료 | p90 ≤ 10s | `link_opened` → `first_vote_cast` |

## 2. 상태 기계

방(`/room/[id]`)은 단일 라우트 위에서 상태만 바뀐다. 화면·버튼은 상태 × 역할 조합으로 결정된다.

```
         ┌──────────────────────────────────────────┐
         │              waiting                      │
         │  · 방장: 후보 카드 + [투표 시작] + URL복사  │
         │  · 참여자: 후보 미리보기(비활성) + "N명 모이는 중" │
         └────────────────────┬─────────────────────┘
                              │
               [방장: 투표 시작] (room_voting_started)
                              │
                              ▼
         ┌──────────────────────────────────────────┐
         │              voting                       │
         │  · 공통: 후보 👍/👎 · 타이머 · 폴링 1s     │
         │  · 방장만: [지금 마감하기]                  │
         │  · 참여자: 마감 버튼 없음(차단)             │
         └────────────────────┬─────────────────────┘
                              │
            ┌─────────────────┼──────────────────┐
     [타이머 0]         [방장: 지금 마감]     [폴링 lazy 확정]
     자동 확정          조기 종료            (deadlineAt 지남 시)
            └─────────────────▼──────────────────┘
                              │
                              ▼
         ┌──────────────────────────────────────────┐
         │              decided                      │
         │  · 공통: 🎉 1등 카드 + 컨페티               │
         │  · 공통: [처음으로] (재시작)               │
         │  · 투표/수정 불가(잠김)                    │
         └──────────────────────────────────────────┘
```

### 상태별 역할 뷰

| 상태 | 방장이 보는 것 | 참여자가 보는 것 |
|---|---|---|
| **waiting** | 후보 5개 카드 · URL 복사 버튼 · [투표 시작] · 현재 입장 인원 | "방장이 곧 시작해요" 안내 · 후보 미리보기(클릭 비활성) · 입장 인원 |
| **voting** | 후보 👍/👎(활성) · 타이머 · 남의 실시간 득표 · [지금 마감하기] | 후보 👍/👎(활성) · 타이머 · 남의 실시간 득표 (마감 버튼 없음) |
| **decided** | 1등 카드 · 최다득표 수 · [처음으로] | 동일 |

## 3. 분기 & 결정 지점

| # | 분기 조건 | 기본 동작 | 설계 의도 |
|---|---|---|---|
| D-1 | **방장이 [투표 시작]을 안 누름** | 대기 상태 유지. 타이머 카운트 시작 안 됨 | 링크 먼저 뿌리고 사람 모인 뒤 시작 가능하게. 빈 방에 타이머 낭비 방지 |
| D-2 | **참여자가 [지금 마감] 시도** | 방장에게만 버튼 노출 — 참여자 UI에 아예 없음 | 마감 권한 남용 및 분쟁 차단. 방장 = 운영자 원칙 |
| D-3 | **타이머 0초 도달** | 폴링 시 `deadlineAt` 경과 감지 → lazy 자동 확정 | 별도 크론 서버 없이 클라이언트 폴링이 확정 트리거. 인프라 단순화 |
| D-4 | **동점 발생** | `votes` 시퀀스(seq) 기준 먼저 해당 점수에 도달한 후보 승 | "랜덤 추첨"보다 결정론적. 수동 개입 불요 |
| D-5 | **후보가 다 별로임 / 빠진 식당 있음** | 투표 중 [+ 후보 직접 추가] 허용 (대기·투표 양 상태) | 콜드스타트 해결 후에도 선택지 보완 허용 → 불만 이탈 방지 |
| D-6 | **방장도 투표하고 싶음** | 방장 = 참여자 화면 + 운영 버튼. 동일하게 👍/👎 가능 | "방장만 소외"되는 이상한 UX 제거. 공정한 합의 강화 |
| D-7 | **참여자가 [투표 시작] 전에 링크 진입** | 대기 라운지(후보 미리보기 비활성)로 안착 | 성급한 단독 투표 방지. "함께 시작"의 의식(ritual) |
| D-8 | **닉네임 중복** | 허용(데모). 같은 닉네임도 별개 참여자로 처리 | 데모 규모에서 가입벽 0 원칙 > 식별 정밀도 |
| D-9 | **한 기기 돌려쓰기(폰 하나로 여러 투표)** | 허용(데모). 동일인 취급 | 현장 데모 편의 > 데이터 정합성 엄밀성 |

## 4. 엣지 케이스 & 대응

| 케이스 | 발생 상황 | 대응 |
|---|---|---|
| **빈 방 (방장 1명, 0표)** | 방 생성 후 아무도 안 들어옴 | 안 깨지고 안내 문구 표시. NSM 분자 제외(실투표자 <2명) |
| **새로고침 (참여자)** | 투표 중 브라우저 새로고침 | `localStorage`에 닉네임·역할 저장 → 재진입 시 자동 복원, 재입력 불요 |
| **링크 받고 관망** | 입장(`joined`)했으나 투표 안 함 | 정상 허용. 아하② 누수 지점으로 별도 계측(joined ≠ vote_cast). 강제 투표 유도 없음 |
| **타이머 만료 — 서버 크론 없음** | `deadlineAt` 지났는데 폴링 응답 없음 | 다음 1초 폴링 도착 시 서버가 `deadlineAt` 경과 감지 → lazy 자동 확정. 늦어도 1초 내 반영 |
| **후보 0개 (목 실패)** | 후보 생성 API 오류 | mock 폴백 5개 강제 노출 (아하① 즉사 방지 철칙 ②4항) |
| **투표 중 후보 추가** | 참여자가 [+ 후보 직접 추가] | 즉시 카드 삽입 + 폴링 반영. 기존 득표 보존 |
| **decided 후 재접속** | 링크로 재입장 | decided 화면 그대로 표시. 투표 불가(잠김) |

## 5. 마찰 지점 & 이탈 리스크

각 단계의 drop-off 위협과 모디의 방어 설계를 퍼널로 정리한다.

```
[랜딩 방문]
    │  ← 이탈 리스크: "또 다른 앱?" 첫인상 불신
    │  → 방어: 한 줄 설명 + CTA 하나("지금 만들기")만. 회원가입 없음.
    ▼
[방 생성 폼 입력]
    │  ← 이탈 리스크: 폼이 길면 포기
    │  → 방어: 필드 5개 최소화 (모임명·위치·인원·예산·타이머 프리셋). 기본값 채워져 있음.
    ▼
[아하① — 후보 5개 표시]                  ← 여기서 방장이 가치를 처음 체감
    │  ← 이탈 리스크: 후보 느리거나 0개 → 방장 이탈, 링크 안 공유
    │  → 방어: 1.5s 로딩 연출 후 mock 5개 즉시. 폴백 보장. (p90 ≤3s 목표)
    ▼
[링크 공유 → 참여자 수신]
    │  ← 이탈 리스크: 링크 클릭 전 "또 가입해야 해?" 인식으로 무시
    │  → 방어: 링크 메시지 자체에 "가입 없음" 맥락 (UX 카피 중요)
    ▼
[참여자 링크 진입 → 닉네임 입력]
    │  ← 이탈 리스크: 닉네임 폼도 마찰. 빈 화면에 혼자 남은 느낌.
    │  → 방어: 닉네임 한 필드만. 입력하면 즉시 대기 라운지.
    ▼
[아하② — 첫 투표 완료]                   ← 여기서 참여자가 가치를 처음 체감
    │  ← 이탈 리스크: 투표 화면 도달 후 관망 (joined ≠ vote_cast 누수)
    │  → 방어: 후보 카드 첫 노출 시 기본 포커스, 👍/👎 큰 터치 영역. 투표 안 해도 무방하나 계측.
    ▼
[투표 진행 — 타이머 중]
    │  ← 이탈 리스크: 타이머가 너무 길면 집중 분산
    │  → 방어: 1/3/5분 프리셋 (데모 기본 1분). 막판 10초 빨강+깜빡 긴장감 부여.
    ▼
[decided — 결과 발표]
    │  ← 이탈 리스크: 결과가 시시하면 재사용 의향 저하
    │  → 방어: 🎉 컨페티 + 1등 카드 zoom-in 연출. "같이 정했다"는 집단 감정 최대화.
    ▼
[재사용 의향 형성]
    ← 이탈 리스크: 결정 완료 후 앱 닫고 끝
    → 방어(P1): [처음으로] 버튼으로 새 방 진입 유도. 🅲 리텐션 루프 시작점.
```

### 퍼널 요약 (이벤트 → 예상 전환율)

| 단계 | 이벤트 | 목표 전환율 | 비고 |
|---|---|---|---|
| 방문 → 방 생성 | 랜딩 → `room_created` | 측정(데모) | 방장 대상, 높아야 함 |
| 방 생성 → 후보 노출 | `room_created` → `candidates_shown` | **≥ 98%** | 아하① 도달률(②) |
| 링크 → 입장 | `link_opened` → `joined` | 측정(데모) | 가입벽 제거 효과 측정 |
| 입장 → 첫 투표 | `joined` → `first_vote_cast` | **≥ 60%** | 아하② 도달률(②) |
| 투표 시작 → 결정 완료 | `room_voting_started` → `room_decided` | **≥ 80%** | Task success KPI(⑤) |

## 6. Flow ↔ 지표 매핑

각 단계가 어떤 이벤트를 쏘고, 어떤 KPI로 이어지는지 한눈에.

| Flow 단계 | 측정 이벤트(⑤) | 관련 KPI / 아하 | 지표 유형 |
|---|---|---|---|
| 방 생성 완료 | `room_created` | 완료율 분모 · 아하① 분모 | 선행 |
| 후보 5개 카드 노출 | `candidates_shown` (latencyMs 포함) | **아하① 도달률** (≥98%) · TTV ≤3s | 선행 · 가드레일 |
| 참여자 링크 클릭 | `link_opened` | Acquisition · 아하② 분모 | 선행 |
| 닉네임 입력 후 입장 | `joined` | 방당 참여자 수 · Referral(≥3명/방) | 선행 |
| 첫 투표 | `first_vote_cast` (= vote_cast seq=1) | **아하② 도달률** (≥60%) · TTV ≤10s | 선행 |
| 이후 투표(반복) | `vote_cast` × N | Engagement(방당 투표 수) | 선행 |
| 후보 직접 추가 | `candidate_added` | 참여 깊이 · 관망 vs 기여 분리 | 보조 |
| 결정 확정 | `room_decided` (byTimer 여부 포함) | **NSM +1** · 완료율 분자 · 소요시간(durationSec) | **핵심 후행** |

### 핵심 파생 지표 (⑤ 계산식)

```
아하① 도달률    = candidates_shown / room_created         목표 ≥ 98%
아하② 도달률    = first_vote_cast (distinct participant) / link_opened  목표 ≥ 60%
결정 완료율      = room_decided / room_created              목표 ≥ 80%
결정 소요시간    = room_decided.ts - room_created.ts        목표 중앙값 ≤ 3분
방당 투표 참여자 = joined distinct / room                  목표 ≥ 3명
NSM(WCD)        = room_decided where 실투표자 ≥ 2명 (주간) 핵심 북극성
```

---

> **다음 문서:** ④ 저니맵은 위 Happy Path를 감정선·시간축으로 펼쳐, 방장의 "총대 스트레스 → 아하① 해방"과 참여자의 "가입 공포 → 아하② 소속감"이 각 단계에서 어떤 감정 파고를 그리는지 시각화한다.
